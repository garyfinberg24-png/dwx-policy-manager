Root Cause #1 — Null Array in PolicyAnalytics Compliance Tab
Analysed 4s ago

Critical — User-Facing Crash

ErrorBoundary triggered 7 times. Users see fallback UI. Component unrecoverable without page reload.

What happened: The renderComplianceTab() method in PolicyAnalytics.tsx:2847 calls .map() on this.state.complianceViolations before the async data load completes. The initial state sets this to undefined instead of [].
Why it happened: Race condition — the compliance tab renders before loadComplianceData() resolves. The state initializer in the constructor sets complianceViolations as undefined (omitted from initial state), not as an empty array.
Pattern detected:
Page load → Tab switch → Crash → ErrorBoundary → Retry → Crash again → Eventually resolves
Suggested fix:
  1. 1
    Initialise complianceViolations as empty array in constructor
    In PolicyAnalytics.tsx constructor, add complianceViolations: [] to the initial state object.
    // PolicyAnalytics.tsx — constructor, line ~180 this.state = { ... // complianceViolations not set (becomes undefined) complianceViolations: [], // Prevent .map() on undefined ... };
  2. 2
    Add defensive guard in renderComplianceTab
    Even with the init fix, add a null guard for defence in depth. This pattern is used in other tabs already.
    // PolicyAnalytics.tsx:2847 const violations = this.state.complianceViolations.map(v => ... const violations = (this.state.complianceViolations || []).map(v => ...

Root Cause #2 — SharePoint API Throttling on PM_Policies
Analysed 4s ago

High — Service Degradation

3 requests throttled (429). Indicates burst of API calls exceeding SharePoint rate limits.

What happened: 24 requests to PM_Policies within 3 minutes triggered SharePoint's rate limiter. The throttled requests returned 429 with Retry-After: 120 headers. Network monitor shows requests are unbatched — multiple components independently query the same list.
Root cause: PolicyHub, PolicyAdmin, and PolicyAnalytics each call PolicyService.getPolicies() independently in componentDidMount(). When navigating quickly between pages, parallel fetches exhaust the request budget.
  1. 1
    Add request deduplication to PolicyService
    Cache in-flight requests so multiple callers share the same promise. This is a common pattern for preventing duplicate concurrent API calls.
    // PolicyService.ts — add request cache private _pendingRequests: Map<string, Promise<any>> = new Map(); private async cachedQuery<T>(key: string, fn: () => Promise<T>): Promise<T> { if (this._pendingRequests.has(key)) return this._pendingRequests.get(key); const promise = fn().finally(() => this._pendingRequests.delete(key)); this._pendingRequests.set(key, promise); return promise; }
  2. 2
    Implement Retry-After header handling
    When a 429 is received, read the Retry-After header and delay before retrying. Add to the existing retryWithDLQ pattern.
