Role Overview & Navigation

ⓘ
Who is the Author?
The Author role is assigned to policy writers — staff responsible for creating, editing, and maintaining policy documents. Authors have all Employee-level access plus the ability to create policies, manage policy packs, and view their authoring dashboard. Authors can draft and edit policies but cannot approve or publish them — that requires Manager or Admin privileges.
⚠
Pre-requisite: The test account must be assigned the Author role by an Admin via Settings > Users and Roles, or be a member of a SharePoint group mapped to the Author role (e.g., PM_PolicyAuthors).

Policy Lifecycle Overview

ⓘ
Understanding the Policy Lifecycle
Before testing, it's critical to understand the status progression of a policy. As an Author, you control the first two stages (Draft and Submit for Review). The remaining stages are handled by Managers and Admins.
Draft → In Review → Approved → Published → Archived
In Review → Rejected → Draft (revision)
StatusWho ActsWhat Happens
DraftAuthorAuthor creates/edits policy content, metadata, and settings. Version starts at 0.1.
In ReviewAuthor → ReviewerAuthor submits for review. Reviewers receive notifications and can approve or reject.
ApprovedManager/AdminReviewer approves. Policy is ready for publishing.
PublishedManager/AdminAdmin publishes. Creates version 1.0+, generates acknowledgements, sends notifications.
RejectedReviewer → AuthorReviewer rejects with reason. Policy returns to Draft for revision.
ArchivedAdminPolicy no longer active. No new acknowledgements generated.

Create a New Policy

🚀
Navigate to: PolicyBuilder.aspx
The Policy Builder is a multi-step wizard that guides the author through creating a new policy. It covers metadata, content, settings, quiz assignment, and review before submission.
AUT-02 Create a New Policy — Wizard Step 1: Metadata P1 Critical
Test the first step of the policy creation wizard — entering basic metadata including title, category, department, and classification.
  1. 1
    Click "Policy Builder" in the navigation
    This navigates to PolicyBuilder.aspx. You should see the Policy Author Enhanced component with a wizard interface.
  2. 2
    Click "Create New Policy" or similar button
    This initialises a new policy wizard. Step 1 should display metadata fields.
  3. 3
    Fill in the required metadata fields
    Complete all required fields:
    • Policy Title — e.g., "Test Data Protection Policy"
    • Category — select from dropdown (HR, IT, Finance, Compliance, etc.)
    • Sub-Category — if available
    • Department — owning department
    • Policy Owner — person picker
    • Data Classification — Public, Internal, Confidential, Restricted, or Regulated
    • Compliance Risk Level — Critical, High, Medium, Low, or Informational
    • Effective Date — when the policy takes effect
    • Review Date — next review due date
  4. 4
    Test field validation
    Try to proceed to Step 2 without filling required fields. Error messages should appear for each missing required field.
  5. 5
    Click "Next" to proceed to Step 2
    With all required fields filled, the wizard should advance to the content editing step.
✓ Expected Results
  • Wizard displays Step 1 with all metadata fields
  • Dropdowns populate with correct options
  • Person picker allows searching and selecting users
  • Validation prevents advancing with empty required fields
  • Policy number auto-generated (e.g., "POL-0001")
  • "Next" button advances to Step 2
AUT-03 Create a New Policy — Wizard Step 2: Content Editor P1 Critical
Test the rich text content editor — the core of policy authoring. Verify formatting tools, template support, and image/infographic insertion.
  1. 1
    Verify the rich text editor loads
    Step 2 should present a rich text editor (similar to Word). The toolbar should include formatting controls: Bold, Italic, Underline, Headings, Lists, Links, Tables, and Image insertion.
  2. 2
    Type policy content
    Write a few paragraphs of test content. Use headings (H1, H2, H3), bullet lists, numbered lists, and bold/italic formatting to verify they work.
  3. 3
    Test template selection (if available)
    If a template selector exists, choose a template (e.g., "Standard Policy", "Image/Infographic Policy"). The editor should populate with the template structure.
  4. 4
    Insert an image or table
    Use the toolbar to insert an image and a table. Verify they render correctly in the editor.
  5. 5
    Click "Next" to proceed to Step 3
    Content should be preserved as the wizard advances.
✓ Expected Results
  • Rich text editor loads with full formatting toolbar
  • All formatting options (bold, italic, headings, lists) work correctly
  • Templates populate the editor with pre-structured content
  • Images and tables render in the editor
  • Content is preserved when navigating between wizard steps
AUT-04 Create a New Policy — Wizard Step 3: Settings & Quiz P2 High
Test the settings step — acknowledgement requirements, read timeframes, distribution scope, and quiz assignment.
  1. 1
    Verify settings fields
    Step 3 should present configuration options:
    • Requires Acknowledgement — toggle on/off
    • Read Timeframe — dropdown: Immediate, Day 1, Day 3, Week 1, Week 2, Month 1, Month 3, Month 6, Custom
    • Distribution Scope — All Employees, Department, Location, Role, Custom, New Hires Only
    • Requires Quiz — toggle, with quiz selector if enabled
    • Retention Category — Standard (3yr), Extended (7yr), Regulatory, Legal, Permanent
  2. 2
    Enable "Requires Acknowledgement" and select a read timeframe
    Toggle acknowledgement on. Select "Week 1" as the read timeframe. This means employees will have 7 days to read and acknowledge the policy.
  3. 3
    Enable "Requires Quiz" and select a quiz
    Toggle quiz on. A quiz selector should appear — select an existing published quiz from the dropdown. If no quizzes are available, note this for the Admin tester.
  4. 4
    Click "Next" to proceed to review
    The wizard should advance to a review/summary step.
✓ Expected Results
  • All settings fields display and are interactive
  • Read timeframe dropdown shows all options
  • Quiz toggle reveals quiz selector when enabled
  • Settings are preserved when moving between steps
AUT-05 Save Policy as Draft P1 Critical
Verify the policy can be saved as a Draft and appears in the author's policy list for later editing.
  1. 1
    Click "Save as Draft" on the review step
    At the final wizard step (or at any step via a "Save Draft" button), save the policy. A success message should appear.
  2. 2
    Verify the policy was created
    Navigate to the Policy Author dashboard (PolicyAuthor.aspx). The newly created policy should appear in your policy list with status "Draft".
  3. 3
    Open the draft and verify data integrity
    Click on the draft policy. All metadata, content, and settings you entered should be preserved exactly as entered.
✓ Expected Results
  • Policy saved with success message
  • Policy appears in Author dashboard with "Draft" status
  • Auto-generated policy number (e.g., POL-0001) is assigned
  • Version shows as 0.1 (initial draft)
  • All entered data (title, category, content, settings) is preserved
  • Record created in PM_Policies SharePoint list

Edit an Existing Policy

AUT-06 Edit a Draft Policy P2 High
Verify an author can open and edit a draft policy they created, and that changes are saved correctly.
  1. 1
    Navigate to Policy Author dashboard
    Go to PolicyAuthor.aspx. Your draft policies should appear in the "Policies" tab.
  2. 2
    Click "Edit" on the draft policy
    Select the draft policy created in AUT-05. Click the edit action (pencil icon or "Edit" button).
  3. 3
    Modify the title and content
    Change the policy title (e.g., append " — Updated"). Edit some content in the rich text editor.
  4. 4
    Save the changes
    Click "Save" or "Save Draft". Confirm with the success message.
  5. 5
    Re-open and verify changes persisted
    Navigate away and come back. Open the policy again — your changes should be there.
✓ Expected Results
  • Draft opens in edit mode with all existing data loaded
  • Changes save successfully with confirmation
  • Changes persist across page reloads
  • Status remains "Draft" after editing
  • Audit log entry created for the update

Submit for Review

⚠
Key Author Workflow: Submitting for review is the primary handoff point. Once submitted, the policy moves to "In Review" status and is locked for editing until the review is complete (approved or rejected). The Author cannot modify the policy while it's under review.
AUT-07 Submit a Policy for Review P1 Critical
Test the submission workflow — submitting a draft policy for manager/reviewer approval.
  1. 1
    Open the draft policy for editing
    Navigate to the policy created in AUT-05 / AUT-06.
  2. 2
    Click "Submit for Review"
    Look for a "Submit for Review" button (may be on the review step of the wizard, or as an action button on the policy card). Click it.
  3. 3
    Select reviewers (if prompted)
    The system may prompt you to select reviewers using a people picker. Select at least one reviewer (use the Manager test account).
  4. 4
    Confirm the submission
    A confirmation dialog may appear. Confirm the submission.
  5. 5
    Verify the status change
    The policy status should change from "Draft" to "In Review". The policy card should update to show the new status with a blue badge.
  6. 6
    Attempt to edit the policy while In Review
    Try to edit the policy. The edit controls should be disabled or the system should prevent changes while the policy is under review.
✓ Expected Results
  • "Submit for Review" action is available for Draft policies
  • Policy status changes to "In Review" after submission
  • SubmittedForReviewDate is set to current timestamp
  • Reviewers receive notification: "Policy submitted for your review"
  • Policy is locked for editing while In Review
  • Audit log records the submission event
AUT-08 Handle a Rejected Policy P2 High
Test the rejection workflow — when a reviewer rejects the policy, the author should be notified and able to revise and resubmit.
⚠
Pre-requisite: Have the Manager tester reject the policy submitted in AUT-07 with a rejection reason (e.g., "Needs more detail in section 3"). This must happen before starting this test.
  1. 1
    Check notifications for a rejection alert
    The notification bell should show a new notification: "Policy Rejected: [Title]" with the rejection reason.
  2. 2
    Open the rejected policy
    Navigate to your Author dashboard. The policy should now show "Draft" status (returned for revision). The rejection reason should be visible.
  3. 3
    Edit the policy to address the feedback
    Open the policy editor. Make changes to address the reviewer's feedback.
  4. 4
    Resubmit for review
    Save changes and submit for review again. The cycle repeats.
✓ Expected Results
  • Author receives rejection notification with the reason
  • Policy returns to "Draft" status and is editable again
  • Rejection reason is visible on the policy detail/card
  • Author can edit and resubmit the policy
  • Resubmission follows the same workflow as initial submission

Policy Packs

🚀
Navigate to: PolicyPacks.aspx
Policy Packs allow authors to bundle multiple related policies together for grouped distribution. For example, a "New Employee Onboarding Pack" might include IT Security, HR Handbook, and Code of Conduct policies.
AUT-09 Create and Manage a Policy Pack P2 High
Test the full lifecycle of a Policy Pack — create, add policies, and save.
  1. 1
    Navigate to Policy Packs
    Click "Policy Packs" in the navigation bar. The Policy Pack Manager should load with a list of existing packs (or an empty state).
  2. 2
    Create a new pack
    Click "Create Pack" or similar. A form or dialog should appear. Enter:
    • Pack Name — e.g., "New Employee Onboarding Pack"
    • Description — brief description of the pack's purpose
    • Target Audience — who this pack is for
  3. 3
    Add policies to the pack
    Use the policy selector to search for and add published policies to the pack. Add at least 2-3 policies.
  4. 4
    Save the pack
    Click "Save". The pack should appear in the pack list with the correct policy count.
  5. 5
    Edit the pack — add or remove a policy
    Open the pack for editing. Remove one policy and add a different one. Save and verify the changes persist.
✓ Expected Results
  • Pack creation form displays with required fields
  • Policy selector shows published policies for adding
  • Pack saves successfully and appears in the list
  • Policy count on the pack card matches the number of policies added
  • Editing the pack preserves existing data
  • Record created in PM_PolicyPacks and PM_PolicyPackAssignments lists

Author Dashboard

🚀
Navigate to: PolicyAuthor.aspx
The Author Dashboard provides a centralised view of all policies you've authored, pending approvals, delegation settings, and recent activity. It has multiple tabs for different views of your authoring work.
AUT-10 Author Dashboard — Tabs & Content P2 High
Verify all tabs of the Author Dashboard load correctly and display accurate data.
  1. 1
    Open the Policy Author page
    Click "Policy Author" in the navigation bar, or go to PolicyAuthor.aspx.
  2. 2
    Verify the tab bar
    You should see tabs for:
    • Policies — your authored policies with status indicators
    • Approvals — pending approval requests on your policies
    • Delegations — delegation settings and history
    • Activity — recent activity log
  3. 3
    Check the Policies tab
    This should list all policies you've created, with columns for: Title, Status (Draft/In Review/Approved/Published), Version, Category, Last Modified date. The draft from AUT-05 should be visible here.
  4. 4
    Check the Approvals tab
    If you have policies in review, they should appear here with the reviewer's name and submission date.
  5. 5
    Check the Activity tab
    Recent actions (created, edited, submitted) should appear in a timeline/log format with timestamps.
✓ Expected Results
  • All tabs render without errors
  • Policies tab shows accurate list with correct statuses
  • Approvals tab shows pending reviews (if any)
  • Activity tab shows a chronological log of recent actions
  • Data is consistent across tabs (no mismatched statuses)
✓

  • Activity log shows recent actions with timestamps
