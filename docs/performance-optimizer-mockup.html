Add Request Deduplication Cache to PolicyService
High Impact
Analysis: PolicyHub, PolicyAdmin, and PolicyAnalytics each call PolicyService.getPolicies() independently in componentDidMount(). When navigating quickly, parallel fetches exhaust the request budget. A shared in-flight cache would coalesce these into a single API call.
// PolicyService.ts — add request deduplication + private _pending: Map<string, Promise<any>> = new Map(); + + private async cachedQuery<T>(key: string, fn: () => Promise<T>): Promise<T> { + if (this._pending.has(key)) return this._pending.get(key); + const p = fn().finally(() => this._pending.delete(key)); + this._pending.set(key, p); + return p; + }
Est. -60% duplicate requests, eliminates 429 throttling
Implement Retry-After Header Handling
Medium Impact
Analysis: When SharePoint returns 429, it includes a Retry-After header (typically 120s). The current retry utility uses fixed exponential backoff which may retry too soon, causing cascading throttling. Reading the header value ensures compliance with SP's backoff guidance.
// retryUtils.ts — respect Retry-After header + const retryAfter = response.headers.get('Retry-After'); + const delayMs = retryAfter + ? parseInt(retryAfter, 10) * 1000 + : initialDelayMs * Math.pow(backoff, attempt);
Prevents cascading 429 failures
Add SP List Index for PM_Approvals
Medium Impact
Analysis: PM_Approvals queries averaging 1,240ms suggest a missing list index. The most common OData filter is $filter=Status eq 'Pending' and AssignedToId eq X. Adding a compound index on Status + AssignedToId should reduce this to ~200ms.
# PowerShell — add index to PM_Approvals Set-PnPField -List "PM_Approvals" -Identity "Status" ` -Values @{Indexed=$true} Set-PnPField -List "PM_Approvals" -Identity "AssignedToId" ` -Values @{Indexed=$true}

Design Notes — For Review

  • Section 1 — Performance Score Card: Circular gauge (0-100) with 5 sub-scores (API Latency, Throttling, Error Rate, Payload, Request Volume). Each sub-score has a progress bar. Scores are calculated from real-time network events captured by the NetworkInterceptor.
  • Optimize All Bar: Green gradient banner showing the total optimizations available and the projected score improvement. Single "Optimize All" button applies all recommended settings at once to PM_Configuration.
  • Section 2 — Optimization Cards: Each detected issue gets a card with: issue description, impact bar (0-100%), one or more tuneable controls (slider for numeric values, toggle for on/off features), and an "Apply" button that saves the setting. Cards sorted by impact descending. Applied cards fade to 75% opacity.
  • Tuneable Parameters:
    • Cache TTL (0-120s): How long to cache SP query results client-side before re-fetching
    • Request Deduplication (toggle): Coalesce in-flight duplicate requests into a single API call
    • Lean Query Mode (toggle): Force specific column selects instead of select('*')
    • Default $top Limit (25-500): Max items per SP query page
    • Max Concurrent Requests (1-10): Throttle outgoing API calls to prevent SP rate limiting
    • Server-side Filtering (toggle): Use OData $filter instead of client-side Array.filter()
  • Before/After Comparison: Side-by-side cards showing current metrics vs projected metrics after applying selected optimizations. Green = improved, red = degraded. Updates dynamically as sliders change.
  • Section 3 — AI Performance Advisor: GPT-4o analyses all network events and recommends code-level changes with diffs. Each recommendation has an "Apply Config" button (saves the tuneable setting) or "Copy Script" (for infrastructure changes like SP indexes). Uses the same event-triage Azure Function mode with a performance-specific prompt.
  • Config persistence: All slider/toggle values save to PM_Configuration via AdminConfigService.saveConfigByCategory('Performance', ...). Services read these on init and adjust behaviour accordingly.
  • What "Apply" actually does: Saves a config key (e.g. Perf.CacheTTL=30) to PM_Configuration. Services like PolicyService read this on initialisation and adjust their query behaviour (add caching, reduce $top, enable dedup). The Event Viewer does NOT modify source code — it adjusts runtime configuration that services honour.
